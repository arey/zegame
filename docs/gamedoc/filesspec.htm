Game Files Specification

In this section I present all the different type of files you can encounter if you go in ZeGame sub-directories. 
    1. Text Files
            >>  ID File (.id)
            >>  Animation Script File (.ina)
      >>  INI File (.ini)
    2. Binary Files
            >>  Sprite Frame settings  File (.spi)
            >>  Decors bitmap settings File (.dec)
            >>  Level Tile Library File  (.LiL)
            >>  Universal Tile Library File  (.LiD)
            >>  Level File  (.niv)         
    3. Resources Files 
      >>  Graphic files
            >>  Sound files
    4. ZeGame Script File (.zfs)


I] Text Files

Those kind of files can be edit with a notepad like tool. Every people who wants to become game or level designer, player, developer, ... can handle those text files with handcare to change game settings, improve game design, add new features, ...

1.      ID File

All the files with .id extension are simple text files that match a number with a name. In ZeGame, they are used by a lot of modules that prefer handle number to string, question of execution speed.

You can find id file to identify a lot of token like :
  • important characters like our hero, the pnjs, main monsters, level boss …
  • background and decoration object  which can interact with their environment,
  • pictures, sound or animations resources,
  • grounded zone of the levels,
  • scripts,
  • ...

Here is an example of the id file World\GameScripts.id :

LocoKiFume              1
CochonKiMange           2
VaseKiSeBrise           3
RobotDesNains           4
Like you could see, each script of the scenario has a unique identifier.


2.      Animation Script File

The .ina files are used both by character and different kind of game object.  It allows you to define the animation of each type of character and object type, for instance for an Ogre, a renegade Dwarf, the Hero, a pig, a little jar or train smoke.

Each ina file is divided in 4 parts as :
  1. the enumeration of the main bitmap frames used by the animation,
  2. configuration of additional frames that recovered previous main frames (optional),
  3. the settings of each animation of the character and
  4. the settings of other animations you could put over another animation (optional).
Each one of those parts is called a section and is divided in section blocks from one or more lines.

Syntax
Each section name is preceded by a # character, is followed by a TAB character and then by the number of section blocks.
In the below instance, the animation is composed from 8 main frames :
 #frame     8

Between each keywords, you have to use TAB character. Your comments must be declared at the beginning of a line and be preceded by //.


2.1)  frame section

You have to enumerate all the bitmap used by this animation. So you have to input the name of the bitmap files that are in the same directory than the .ina file, but without their extension. Before the short name of the bitmap, you have to put the characters F:

Notice that all frame is made of a fixed number of sprite and you have to do attention of the order of your enumeration. Actually, in the following sections, when you specify the index of a sprite, it doesn’t start from the beginning of its frame but from the beginning of the first frame !

Example:

F:HeroJump1
In the Persos\Hero\ folder, you have the HeroJump1.bmp that is used to make the animation of our Hero.
2.2) frameSuperposee section
You have to configure all the additional frames that you could put over another animation you have done in the frame section.

So you have to enumerate all the bitmap files (S:BitmapFile). Then link them with a previous main frame that they will superimposed (F:MainFrameBitmapFile). After, you have to choose what for kind of sprite type this frame is (T:TypeObjSprite); this is an ID that you have previously set in the Persos\IDTypeObjSprite.id id file. Another step is to choose where the sprites from this frame will be display : for a character, you can choose a member of his body (M:DisplayArea). Finally, you must specify if this frame will be load in Ram at the beginning of the game or just when we will need it.

Example:

S:HeroJmpEpe1      F:HeroJump1      T:EPEE1      M:DANS_MAIN_DROITE      E:MEM

The HeroJnmpEpe1.bmp file contains the sprites of a sword (EPEE1) that can be map on the DANS_MAIN_DROITE Hero when he jumps (so when we used sprites of  HeroJump1.bmp). This file will be load in memory when the game will start.

2.3)    anim section
In this section, you must define all the animations of a character or another object. A character can have a lot of animations used when he walks, runs, jumps, fights, swims, …

An animation has a name you define by N:AnimationName. AnimationName is an ID of an animation that you must previously define in the id file Persos\IDAnimations.id.
All the animations are made of a fixed number of sprites, that you can precise by N:SpriteNumber.
There are several ways to set this section. You must indicate the type by T:TypeNumber.

So the general syntax for an anim section bloc is :
N:Name      T:Type            C:SpriteNumber      I:Indexes


Let’s see the different type of animation settings that you can encoutered :

·        type 1 is the simplest form to set an animation because you just need to give the index of the first sprite of the animation by I:SpriteIndex.

·        with type 2 you can precise for each of the 8 points of view, the index of the sprites that will use. Here is the syntax : I:IndexE1, IndexE2,… IndexEn; IndexSE1, IndexSE2, …IndexSEn; … ; IndexNE1, IndexNE2,…,IndexNEn.

·        type 3 is like type 1, but for just 1 cardinal point.

·        and type 4 is like type 2, but for just 1 cardinal point.

·        type 0 is a particular type that associate an animation to one sprite that is a tile. So you must give its ID like this : I:TileID you have already set up in World\IDTiles.id

Note that types 1 and 2 are for animations that require 8 cardinal points and type 3 and 4 are used in the animations that have just 1 position. And as I  told you in the frame section, this index is taken from the first Frame you enumerate.

Examples:

N:MARCHE1      T:1      C:16      I:0
It describes a walk animation that is build with the 16 first sprites of all the frames.

N:BRISE2      T:3      C:27      I:16

Same thing with a vase that breaks and that we can look just for one view.

N:DEMITOUR1      T:2      C:4      I:64,80,96,112;80,96,112,0;96,112,0,16;112,0,16,32;0,16,32,48;16,32,48,64;32,48,64,80;48,64,80,96

It’s the animation of a character that wants to do a half-turn in 4 sprites. To do that we take the sprites of the character that walks.


N:REBUILD1      T:4      C:16      I:31,30,29,28,27,26,25,24,23,22,21,20,19,18,17,16

Same thing with a vase that rebuilds and that we can look just for one view.

 

N:CASSER1      T:0      C:1      T:BOCALBRK

A vase broken has just one sprite : it’s the sprite based on the BocalBrk tile.

3.      INI File

Classic microsoft windows .ini files. They are used to store general settings of the game the player can change. And are also used in some cases to store more valuable settings. In this case, they nay be encrypted or/and included in a .pak file.
An example of real .ini file is ZeGame.ini where all the main settings of Nains are stored.

II] Binary Files

Those file formats can not interested you. They are here more for potential coders that would like change some datas of the game with hexadecimal editor or for possible developers that wish update the game. So they are more fixed game datas that are handle by the multiple editors.

1.      Sprite frame settings File

Generated when you create a frame with AnimEditor. This frame contains one or more sprites. Each one has a cartesian position on this frame. This (X,Y) position has been taken since left top corner of the frame.
The extension of those files is .spi

Header :
    • Name of this .spi file du fichier .spi (12 characters string),
    • Number of sprites in this frame (int),
    • Index of the transparent background color in the 256 colors frame (int),
    • Index of the shadow color (int),
    • RGB component of previous transparent background color (24 bits RGB format),
    • RGB component of previous shadow color (24 bits RGB format).

Body :

Then for each sprites of this frame :

  • Numero of this sprite (int),
  • Its dimention in pixel (Width, Height) (2 int),
  • Its position in pixel (X,Y) in the frame (2 int),
  • Its center of gravity (X,Y) (2 int).

    2.      Decors bitmap settings File

    Generated when you create a new background scenery from a simple tile bitmap (.bmp) file with the CreateDecors util.

    When you design a level, you can place those scenery objects on the map. To do that we need to know some basic information about this bitmap : is it a ground tile or an object tile ? does it project shadow or not ? what are the parts of this bitmap that reach the ground ? ...

    The extension of those files is .dec

    Header :

      • Version of this .dec file (1 float). Presently, we are at the version 0.98,
      • Unique Identication number of this tile (int),
      • Its dimention in pixel (Width, Height) (2 int),
      • RGB component of transparent background color (24 bits RGB format),
      • Index of this transparent background color (int),
      • Number of diamond-shaped that composed this scenery (int),
      • Position in pixel (X,Y) of the main diamond-shaped.

    Body :

    • Then for each diamond-shaped of this tile :
  • Relative position (dU, dV) of this trapezoid with regard to the main one (2 int),
  • Height (in pixel) of the higher point of this trapezoid (int),
  • Floor type of this diamond-shaped : sand, rocks, root vegetables, grass ... (char),
  • Collision type : none, partial or total (int),
  • Boolean that tell if it is used to draw a part of the tile (bool),
  • Rectangle coordinates of this sheet of tile (4 int).
    •
    • Position in pixel (X,Y) of the top left corner of this bitmap in the CreateDecors view (2 int),
    • Boolean that indicate if we need to draw this tile in several times or in one time (bool),
    • Boolean that indicate if there are circle collisions with this object (bool),
    • Position in pixel (X,Y) of the circle collision center (2 int),
    • Radius in pixel of this circle (int),
    • Special display feature : transluence, shadow ... (int).

    5.      Level File

    Those files are the main files (with the level tile library .LiL file) of a level. When you load (or save) a game level, this is the file corresponding to the level name that is loaded (or saved). They are stored in the Levels\ sub-directory.

    The extension of all those files is .niv

    Header :

    • Version number of this file : actually we are at v2.5 (1 float),
    • Name of the level map (15 chars),
    • ID numner of this level (int),
    • Dimension (Width x Height) in number of case (2 int),
    • Name of the level tile library (.LiL) associated to this level (15 chars),
    • Level number of floor (int),
    • Number of game object (tile) present on this level (int),
    Body :
    • Then for each game object :
    • ID number of this object (int),
    • Position (U,V) on the map (2 int),
    • Floor wich it belongs (int),
    • Layer on this floor (ground, sky, ...) wich it belongs (int),

    • GameObject based class information :
      • ID number (int),
      • Type : scenary object, character, background object ... (int),
      • ID number allocation type : automatic, predefined ... (int),
      • Current states of the GameObject :
        • number of state items (int),
        • list of states (n int),
      • Current memory of the GameObject :
        • number of memory items (int),
        • list of memories (n int),
      • Script ID associated to this GameObject (int),

    • Map zones information :
      • Number of zones (int),
      • For each zone : .
        • Type of the zone : circle, rectangle, ... (int),
        • ID number (int),
        • Name of this zone (64 chars),
        • Number of GameObject associated to this zone (int),
        • ID Number list of those GameObject (n int),
        • Information that change with the specific zone

    IV] ZeGame Script File

    Those files are very useful to handle with behaviors of as well characters and objects as scenaritic outcome. They are just text files you can edit with an text editor, their extension are .zfs and have the appearence of simple .ini files.

    If you want to design our own level or just change the behavior of a character, you can edit the file and add (or change) what you want. Be sure to respect the zfs file syntax described below because I don't log the errors when I parse the file, and a mistake can easily crash the game, so be crareful !

    They are made up of a header section and a set of script line section, depending of their complexity.

    1. Header section

      This section has 3 labels :
      • ScriptName : name of this script (same name as the file)
      • ScriptType : this label can take 2 values. The first is MODELE (that means template in english) or TEMPLATE that indicate this script will not be directly used by a game object, but will help to design new script. The second one is PERSONNALISE or CUSTOMIZED that tells that it could be directly used by any object it designed for.
      • ScriptPatron : give the name of the script that it inherits. It could be a template or not. If it has no parent script, just write <none> in this section.

      Here is an example of script header that you can find in RobotDesNains2.zfs :

      [HEADER]
      ScriptName = RobotDesNains2
      ScriptType = PERSONNALISE
      ScriptPatron = Robot
      The script name is RobotDesNains2. This script is used by an object of the game and inherits from the Robot script.


    2. Script Line section

    A script is composed of one or more script line. A script line is always defined by :

    • a trigger condition : the objects which this script is assigned must receive an event of this kind of trigger,
    • a cause line : this script will run only if all those causes are true,
    • an action line : actions that execute this script line if causes are valid.

    An example of zfs script sections could be find in RobotDesNains.zfs :

    [SCRIPT_LINE_1]
    Trigger    = PERSO_DSZONE
    Conditions = { Random(2) & DontKnow(ROBOT_DONNE_COUP_AU_HERO) &
                   DontKnow(ROBOT_DEAD) }
    Actions    = { NewState(DONNECOUP1) & NewMemory(ROBOT_DONNE_COUP_AU_HERO) }

    [SCRIPT_LINE_2]
    Trigger    = EVERY_TIME
    Conditions = { Know(ROBOT_DONNE_COUP_AU_HERO) }
    Actions    = { Sleep(1800) & DelMemory(ROBOT_DONNE_COUP_AU_HERO) }

    [SCRIPT_LINE_3]
    Trigger    = EVERY_TIME
    Conditions = { Know(ROBOT_DEAD) & DontKnow(NAIN_SORTI) }
    Actions    = { Sleep(1800) & NewState(SORTIR1) & NewMemory(NAIN_SORTI) }

    [SCRIPT_LINE_4]
    Trigger    = EVERY_TIME
    Conditions = { DontKnow(ROBOT_DEAD) & Random(200) }
    Actions    = { Sleep(500) & NewState(STAND1) }

    a) Trigger

    • SCN_TRIGGER_EVERY_TIME : script line that is estimate at each frame,
    • SCN_TRIGGER_SCENARIO_LEVEL_INIT : event triggered when a game level start that allow each object to initialize,
    • SCN_TRIGGER_PLAYER_UTILISER : event sent to an object when a character use something on somebody else,
    • SCN_TRIGGER_PLAYER_FRAPPER : event sent to an object when something or someone hit this object,

    • SCN_TRIGGER_PERSO_DSZONE : triggered each time a character is into a ground zone,
    • SCN_TRIGGER_PERSO_ZONEIN : inform that a character has just entered into a ground zone,
    • SCN_TRIGGER_PERSO_ZONEOUT: fired each time a character exit from a gound zone,
    • SCN_TRIGGER_PERSO_DEAD : script line analyzed when a character just died,

    • SCN_TRIGGER_METEO : event sent each time the meteo change,
    • SCN_TRIGGER_TIME : fire when time periods changes (hour, month, year, ...),
    • SCN_TRIGGER_PNJ_DEAD : event triggered when a NPC just dead,
